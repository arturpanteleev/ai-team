# Design: verification-enforcement

## 1. Verdict-контракт

### Канал вердикта
Вердикт живёт в основном артефакте агента (человекочитаемо и машиночитаемо одновременно), парсится **line-anchored** регулярным выражением — это исключает ложные срабатывания на цитаты внутри прозы (главная слабость старого `strings.Contains`):

```
^\*\*Verdict:\*\*\s+(APPROVED|CHANGES_REQUESTED|REJECTED)\s*$
^\*\*Result:\*\*\s+(PASS|FAIL)\s*$
```

Первое совпадение по файлу выигрывает. Харнесс сканирует все `.md`-выходы этапа; берётся первый ненулевой вердикт.

Негативные вердикты: `REJECTED`, `CHANGES_REQUESTED`, `FAIL`. Позитивные: `APPROVED`, `PASS`. Отсутствие вердикта — `none` (не негативный: у analyst/architect/coder вердиктов нет).

### Канал BLOCKED
BLOCKED — отдельный side-channel, потому что заблокированный агент по определению не может создать свои обычные выходы. Агент создаёт `{feature}/status/{agent}.md`:

```
**Status:** BLOCKED
**Blocker:** <причина одной-двумя строками>
```

Харнесс проверяет этот файл **до** stat-проверки выходов. Если найден — этап получает статус `blocked`, выходы не проверяются, пайплайн останавливается (exit-код 2, подсказка `--retry-from {agent}`).

### Служебная секция промпта
Требования контракта (куда писать вердикт, как сигналить BLOCKED, куда писать stage-summary, effort) добавляет **харнесс** в `buildPrompt` единой секцией «## Служебные требования» — а не дублируются в 7 промптах. Ролевые промпты содержат только ролевую специфику (какие значения вердикта допустимы и по каким критериям). Это гарантирует, что контракт промпта и парсер меняются в одном месте и не разъезжаются; на пару «промпт-инструкция ↔ парсер» пишутся contract-тесты.

## 2. Enforcement

Порядок обработки после завершения этапа:

```
Execute → blocked-check → outputs-check → git-guard → verdict-parse →
  ├─ negative verdict:
  │    interactive + loopback доступен → предложить retry (Y/n/diff)
  │    иначе → политика on_negative_verdict (default: stop)
  ├─ gate_after / by_confirm / gate_before(next) — как раньше
  └─ continue
```

- `on_negative_verdict: stop | ask | continue` (per-agent, default `stop`). `ask` в неинтерактивном режиме ведёт себя как `stop`. `continue` — осознанный opt-out (например, для advisory-ревью).
- Loopback: цель — ближайший предшествующий агент `coder` (имя из `loopback_to`, default `coder`); `max_retries` читается из конфига **цели** (соответствует спеке workflow-loopback); при retry цель получает выходы вердикт-агента (review.md) дополнительным входом; при исчерпании retries — остановка с ошибкой «Превышен лимит retries (N) для {target}».
- Неинтерактивный режим: гейты/confirm пропускаются (как раньше, по спеке pipeline-gates), но **негативный вердикт и BLOCKED всегда останавливают** — это и есть детерминированная гарантия для CI.

Alternative considered: `verdict.json` рядом с артефактом (чистый машинный канал). Отклонено на этом этапе: дублирует данные, требует от LLM писать два файла согласованно; line-anchored маркер в одном артефакте проще и достаточно надёжен. Если практика покажет обратное — контракт изолирован в `pkg/verdict`, замена канала локальна.

## 3. Exit-коды и сигналы

| Код | Значение |
|-----|----------|
| 0 | Пайплайн выполнен |
| 1 | Ошибка этапа / негативный вердикт / ошибка инфраструктуры |
| 2 | BLOCKED (нужно вмешательство человека) |
| 3 | Пользователь остановил на gate/confirm/loopback-отказе |

Старая спека pipeline-blocked-status содержала внутреннее противоречие (blocked = «код 0, не ошибка» и одновременно «код, указывающий на BLOCKED»; в CI — 1). Унифицировано: **blocked → 2 всегда** — CI и скрипты различают исходы без парсинга stderr.

`cmd/ai-team run` использует `signal.NotifyContext(SIGINT, SIGTERM)`; при отмене пайплайн генерирует итоговый отчёт и summary (сейчас при ctx.Done отчёта нет).

## 4. Таймауты

- `stage_timeout: 30m` — глобально в config.yaml (генерируется init-ом); `timeout: 45m` — per-agent override; пустое значение → глобальный; `stage_timeout` отсутствует в конфиге → без таймаута (обратная совместимость для существующих конфигов).
- Реализация: `context.WithTimeout` вокруг `Runtime.Execute`; по таймауту этап падает с ошибкой «этап {agent} превысил таймаут {d}».

## 5. Model / Effort

- `model` → аргумент CLI. Таблица инвокации per-CLI: для `opencode` — `opencode run -m <model> <prompt>`. Для неизвестных CLI model не передаётся (warning в консоль) — лучше честный пропуск, чем сломанный вызов.
- `effort` → строка в служебной секции промпта («Уровень усилий: high — потрать больше итераций на проверку»). Универсально для любого CLI; флаг у opencode отсутствует.
- Значение `model: auto` признано «не задано» (историческое значение из init-конфигов) — не передаётся.

## 6. Наблюдаемость

- **Логи**: `AgentCLIRuntime` пишет stdout/stderr агента в `.ai-team/logs/{feature}/{agent}.log` (append при loopback-повторах, с разделителем запуска) и дублирует в консоль через `io.MultiWriter`.
- **SQLite**: `cmdRun` открывает `{target}/.ai-team/web.db` (создаёт при отсутствии), создаёт запись `pipeline_runs` при старте и закрывает её при завершении; этапы пишутся через `StoreRecorder` (реализация нового интерфейса `pipeline.Recorder`: `RunStarted/StageStarted/StageFinished/RunFinished`). Сломанный `WebNotifier` (hardcoded run_id=0) удаляется. WebSocket-hub остаётся для in-process событий сервера; живые обновления фронтенда обеспечиваются поллингом активных runs (интервал 5s) — межпроцессный push вне scope.
- В таблицу `stages` добавляется колонка `verdict` (миграция `ALTER TABLE ... ADD COLUMN` с игнорированием ошибки «duplicate column»).

## 7. Безопасность web

`GET /api/artifacts/*` обслуживает файлы **только** внутри корня артефактов (`--artifacts`, default `.ai-team/artifacts`): путь резолвится через `filepath.Abs` + проверка префикса корня. CORS `*` и `CheckOrigin: true` заменяются на same-origin (dev-фронт ходит через vite proxy — CORS не нужен). Graceful shutdown по SIGINT (требование web-http-server).

## 8. Что сознательно НЕ делается в этом change

- Полная state-machine пайплайна и Interaction-абстракция реализуются прагматично (выделение методов + инжект runtime-фабрики и promptFunc для тестируемости), без полной переписи.
- Межпроцессные live-события CLI→web (нужен IPC/файловый watcher) — отложено.
- Харнесс-запрет push в default-ветку реализуется на уровне промпта deployer (шаг создания ветки) — детерминированная проверка ветки требует знания git-стратегии проекта и откладывается.
- `LLMRuntime` остаётся заглушкой.
