## Context

Обе находки — про один и тот же класс дефекта: контроллер уже знает нужную
информацию (что `Load()` упал; что прошлый run довёл фичу до delivery), но не
доносит её до пользователя в момент, когда это дешевле всего исправить.
Ни одна из них не про потерю данных или нарушение delivery-гарантий.

## Goals / Non-Goals

**Goals:**

- `ai-team list` не должен молча терять агентов из-за ошибки загрузки.
- `ai-team run` должен объяснять, если `--feature` уже был доставлен, а не
  заставлять пользователя реверс-инжинирить это из непонятной ошибки coder.

**Non-Goals:**

- Не вводится запрет на повторное использование `--feature` — это осталось бы
  ломающим изменением для легитимного сценария «доставить фичу, потом
  добавить ещё работы под тем же именем».
- Не меняется delivery enforcement, require-diff guard или verdict-контракт.
- Не создаётся общий "notification" framework — оба предупреждения простые,
  локальные, в stderr.

## Decisions

### Decision: `List()` возвращает failures явно, а не логирует их сама

Альтернатива — логировать ошибку прямо внутри `Registry.List()` — отклонена:
`pkg/agent` не должен решать, КАК сообщать об ошибке (stderr? UI-цвет?
структурированный JSON для будущего `--json` вывода?) — это ответственность
вызывающего кода. `Registry` остаётся чистым API без side-effect'ов
вывода/логирования, только возвращает данные.

### Decision: предупреждение о повторной доставке — warn, не block

Рассмотрена альтернатива — жёстко запретить `run` без `--retry-from`, если
`FindDelivered` находит прошлую доставку. Отклонено: это ломает легитимный
workflow «фича done, но нужно продолжить работу под тем же именем» без
какой-либо пользы — require-diff guard и так не даст доставить пустой diff
дважды. Правильный fix — объяснить, а не запретить.

### Decision: `FindDelivered` — best-effort scan evidence, а не git-branch lookup

Альтернатива — проверять существование delivery-ветки в git — отклонена:
ветка может быть удалена/переименована после merge, а evidence (`run.json` +
`attempts/*/manifest.json`) — источник истины самого контроллера про то, что
произошло. Повреждённые/нечитаемые run директории пропускаются молча внутри
`FindDelivered` — это diagnostic best-effort сигнал для CLI, а не проверка,
влияющая на delivery decisions контроллера.

## Risks / Trade-offs

- `FindDelivered` сканирует все run-директории проекта при каждом свежем
  `run` — на больших количествах run (сотни+) это добавляет заметную, но
  ограниченную задержку перед стартом pipeline. Не оптимизировано (нет
  индекса/кэша), так как проверка выполняется один раз за запуск, а не в
  горячем пути.
- Предупреждение — не гарантия: если evidence для прошлой доставки была
  удалена или run root указывает на другой `--target`, предупреждение не
  сработает. Это осознанно допустимо — это diagnostic UX improvement, а не
  новая delivery guarantee.

## Migration Plan

Оба изменения аддитивны для пользователя (новый stderr-вывод), кроме
внутреннего изменения сигнатуры `Registry.List()` — единственный вызывающий
код в репозитории обновлён в этом же change. Нет пользовательских данных для
миграции.
