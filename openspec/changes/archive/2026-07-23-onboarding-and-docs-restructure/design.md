## Context

README на 2026-07-23 линейно перечисляет установку, quickstart, полное
описание конвейера, полное описание deployer internals, полное описание
evidence layout, конфигурацию, evals и security posture — без разделения на
"нужно для первого запуска" и "нужно, если копаете внутрь". Пользователь
описал результат как "смесь какой-то архитектуры, команды" — то есть жалоба не
на неточность контента (аудит подтвердил, что технические утверждения README
корректны), а на то, что читателю негде остановиться на "достаточно, чтобы
начать работать".

## Goals / Non-Goals

**Goals:**

- Один линейный путь для нового пользователя: от "что это" до "я поставил
  первую фичу", без необходимости заранее знать внутренние термины.
- Глубокое техническое описание остаётся полным и точным, но переезжает туда,
  где его целенаправленно ищут (`docs/ARCHITECTURE.md`), а не встречается по
  пути к Quickstart.
- OpenSpec-цикл разработки документирован для человека отдельно от
  агент-специфичных инструкций Claude Code.
- Минимальный автоматический regression guard, чтобы будущие изменения не
  тихо сломали глоссарий, ссылку между файлами или структуру секций.

**Non-Goals:**

- Не меняется ни одно поведение системы — это чисто документационный change.
- Не переписываются `AUDIT.md` или `INDEPENDENT_AUDIT_2026-07-23.md` — README
  только ссылается на них.
- Не создаётся doc-site/генератор — только markdown в репозитории.
- `docs/docs_test.go` — не полноценный doc-linter (не проверяет орфографию,
  битые внешние URL или качество прозы), а дешёвый grep-guard на структуру.

## Decisions

### Decision: порядок секций README

Что это / Для кого (и для кого не) / Предварительные требования и установка /
Быстрый старт / Как поставить фичу целиком / CLI-справочник / Конвейер и зоны
ответственности (кратко + ссылка) / Конфигурация / Evals / Глоссарий / Граница
безопасности (кратко + ссылка на AUDIT.md) / Разработка (ссылка на
CONTRIBUTING.md).

Альтернатива — оставить полное описание deployer/evidence internals в README
"для полноты" — отклонена: это и есть исходная причина жалобы ("смесь
архитектуры и команд"), и создаёт ложное впечатление, что эти детали нужны для
первого успешного запуска.

### Decision: docs/ARCHITECTURE.md как единственное место глубокого описания

Контент про детерминированный контроль, evidence layout, deployer internals и
layered agent registry перенесён по существу дословно (адаптированы только
вводные фразы и добавлена карта пакетов), без попытки его сократить или
переформулировать — риск случайно исказить уже проверенные аудитом
технические утверждения выше пользы от редактуры формулировок.

### Decision: CONTRIBUTING.md как human-facing OpenSpec guide, отдельно от CLAUDE.md

`CLAUDE.md` остаётся источником истины для Claude Code agent behavior (gate
rule, разрешённые инструменты, маппинг на конкретные tool calls).
`CONTRIBUTING.md` пересказывает тот же OpenSpec-цикл для человека или другого
агента (например, контрибьютора через OpenCode CLI), плюс make-таргеты и
пример добавления built-in агента, которых в `CLAUDE.md` нет и не должно быть
(это не relevant для agent tool-mapping).

Альтернатива — слить оба файла в один — отклонена: они обслуживают разные
execution contexts (машинно-читаемые agent-инструкции конкретного инструмента
vs. человеко-читаемый процесс-гайд) и разные жизненные циклы (CLAUDE.md может
меняться при смене tool-конвенций Claude Code, не меняя сам цикл разработки).
Дублирование описания цикла между двумя файлами — осознанный trade-off,
смягчённый тем, что `docs/docs_test.go` проверяет присутствие всех стадий
цикла в `CONTRIBUTING.md`, а `CLAUDE.md` не является тестируемым артефактом
(agent-инструкции не парсятся кодом).

### Decision: лёгкая, но проверяемая спецификация вместо чистой prose-конвенции

Добавлена capability `project-documentation` с scenario-требованиями к
структуре README/CONTRIBUTING/ARCHITECTURE и сопровождающий
`docs/docs_test.go`, включённый в стандартный `go test ./...` / `make test`.
Это не полная doc-linting система, а минимальный regression guard: дешевле
написать один Go-тест с grep по обязательным заголовкам/терминам, чем
полагаться на то, что ревьюер каждого будущего PR вручную перечитает весь
README целиком.

Альтернатива — просто "работать по договорённости", без формального spec —
отклонена явно по инструкции пользователя: любая продуктовая фича должна иметь
change и обновлять общую спецификацию.

## Risks / Trade-offs

- Дублирование описания OpenSpec-цикла между `CLAUDE.md` и `CONTRIBUTING.md`
  может разойтись со временем, если один файл поправят, а другой забудут.
  Смягчение: `docs/docs_test.go` проверяет присутствие тех же ключевых стадий
  в `CONTRIBUTING.md`; расхождение в деталях (а не в наборе стадий) остаётся
  риском review, а не автоматической проверки.
- grep-тест на заголовки/термины хрупкий к чисто косметическим
  переформулировкам (например, если кто-то решит переименовать секцию) — это
  осознанный компромисс: тест дешёвый и его назначение — ловить случайное
  удаление, а не диктовать формулировки.
- Перенос контента в `docs/ARCHITECTURE.md` меняет относительные ссылки
  (`#deployer-и-canonical-delivery-plan` и `#layered-agent-registry`,
  зависящие от GitHub-слагификации заголовков) — оба anchor проверены вручную
  и покрыты `TestArchitectureDocCoversReferencedAnchors`.

## Migration Plan

Однократное переструктурирование в одном PR: README переписывается полностью,
новые файлы добавляются одновременно, чтобы не было промежуточного состояния с
битыми ссылками. Пользовательских данных или API для миграции нет.
