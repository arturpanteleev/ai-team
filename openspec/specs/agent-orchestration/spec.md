## ДОБАВЛЕННЫЕ Требования

### Требование: Порядок выполнения пайплайна
Система ДОЛЖНА выполнять агентов в строгом порядке, указанном в конфигурации пайплайна.

#### Сценарий: Последовательное выполнение
- **КОГДА** пайплайн запускается (полный запуск)
- **ТОГДА** первый агент в конфигурации выполняется первым
- **И** каждый последующий агент выполняется только после успешного завершения предыдущего
- **И** агенты из конфигурации (переопределяемые через `pipeline:`) ДОЛЖНЫ выполняться в заданном порядке

#### Сценарий: Сбой пайплайна
- **КОГДА** любой агент возвращает ошибку
- **И** НЕ указан флаг `--retry-from`
- **ТОГДА** пайплайн ДОЛЖЕН остановиться
- **И** последующие агенты НЕ ДОЛЖНЫ выполняться
- **И** ДОЛЖЕН быть сгенерирован итоговый отчёт с указанием упавшего этапа

### Требование: Коммуникация через артефакты
Агенты ДОЛЖНЫ общаться через файлы в `.ai-team/artifacts/`.

#### Сценарий: Агент читает входные данные
- **КОГДА** агент запускается
- **ТОГДА** система ДОЛЖНА прочитать пути его входных артефактов из `def.yaml`
- **И** проверить, что каждый входной файл/директория существует (stat)
- **И** вывести в лог: имя артефакта, полный путь, дату изменения, размер
- **И** передать содержимое файловых артефактов в runtime как `[]Artifact`
- **И** ЕСЛИ входной артефакт не найден → пайплайн ДОЛЖЕН остановиться с ошибкой

#### Сценарий: Агент записывает выходные данные
- **КОГДА** агент успешно завершается
- **ТОГДА** система ДОЛЖНА проверить, что его выходные артефакты существуют (stat)
- **И** вывести в лог: имя артефакта, полный путь, дату изменения, размер
- **И** добавить их к общему списку артефактов для следующего агента
- **И** ЕСЛИ выходной артефакт не создан → пайплайн ДОЛЖЕН остановиться с ошибкой

### Требование: Изоляция фич
Каждая фича ДОЛЖНА иметь своё дерево артефактов в `.ai-team/artifacts/{feature}/`.

#### Сценарий: Несколько фич
- **КОГДА** пользователь запускает пайплайн для "add-auth", а позже для "add-payment"
- **ТОГДА** артефакты ДОЛЖНЫ храниться раздельно: `.ai-team/artifacts/add-auth/` и `.ai-team/artifacts/add-payment/`

### Требование: Настройка пайплайна
Пайплайн ДОЛЖЕН настраиваться через `.ai-team/config.yaml`.

#### Сценарий: Кастомный пайплайн
- **КОГДА** пользователь указывает `pipeline: [analyst, coder, deployer]` в конфиге
- **ТОГДА** система ДОЛЖНА выполнить только указанных агентов в заданном порядке

### Требование: Runtime интерфейс с передачей артефактов
Runtime ДОЛЖЕН принимать на вход список верифицированных артефактов.

#### Сценарий: Сигнатура Execute
- **КОГДА** runtime запускает агента
- **ТОГДА** Execute ДОЛЖЕН принимать `inputs []Artifact`
- **И** Artifact содержит: Name, Path, Size, ModTime
- **И** runtime САМ решает, как передать артефакты агенту (встроить в промпт, передать флагом и т.д.)

### Требование: Структура Artifact
`Artifact` — это значение, описывающее переданный файл.

#### Сценарий: Поля Artifact
- `Name` — логическое имя артефакта (из `def.yaml`: "spec", "design", "review")
- `Path` — полный путь к файлу на диске
- `Size` — размер в байтах
- `ModTime` — время последнего изменения

### Требование: Генерация HTML-отчётов в пайплайне
Пайплайн ДОЛЖЕН генерировать HTML-отчёты после каждого этапа и итоговый сводный отчёт.

#### Сценарий: Отчёт генерируется после каждого агента
- **КОГДА** агент завершает выполнение (успешно или с ошибкой)
- **ТОГДА** пайплайн ДОЛЖЕН вызвать генерацию HTML-отчёта для этого агента
- **И** передать в отчёт: статус, время, входы, выходы, ошибку (если есть)

#### Сценарий: Итоговый отчёт в конце
- **КОГДА** пайплайн завершается
- **ТОГДА** пайплайн ДОЛЖЕН вызвать генерацию итогового сводного HTML-отчёта
- **И** отчёт ДОЛЖЕН содержать сводку по всем этапам с навигацией

### Требование: Уведомления в пайплайне
Пайплайн ДОЛЖЕН вызывать Notifier после каждого завершённого этапа.

#### Сценарий: Notify вызывается после каждого этапа
- **КОГДА** агент завершает выполнение
- **ТОГДА** пайплайн ДОЛЖЕН вызвать `Notifier.Notify()` для всех зарегистрированных notifier-ов
- **И** передать StageResult: имя агента, статус, сообщение, время

### Требование: Прогресс-бар в пайплайне
Пайплайн ДОЛЖЕН обновлять строку состояния перед и после каждого этапа.

#### Сценарий: Обновление прогресс-бара
- **КОГДА** пайплайн начинает выполнение агента
- **ТОГДА** прогресс-бар ДОЛЖЕН отобразить имя проекта, фичи и текущего агента
- **И** после завершения агента прогресс-бар ДОЛЖЕН обновиться

### Требование: Ролевая конфигурация в пайплайне
Пайплайн ДОЛЖЕН использовать индивидуальные настройки (model, effort, cli) для каждого агента.

#### Сценарий: Агент запускается с индивидуальными настройками
- **КОГДА** config.yaml содержит model/cli/effort для конкретного агента
- **ТОГДА** пайплайн ДОЛЖЕН передать эти настройки при запуске агента
- **И** НЕ ДОЛЖЕН использовать глобальные значения для этого агента

### Требование: Retry-from в пайплайне
Пайплайн ДОЛЖЕН поддерживать частичный запуск, начиная с указанного агента.

#### Сценарий: Пропуск этапов при retry-from
- **КОГДА** пайплайн запущен с флагом `--retry-from <agent>`
- **ТОГДА** пайплайн ДОЛЖЕН пропустить всех агентов до указанного
- **И** проверить наличие артефактов пропущенных этапов
- **И** ЕСЛИ артефакты отсутствуют — вывести ошибку и остановиться


### Требование: Git diff guard в пайплайне
Пайплайн ДОЛЖЕН проверять git diff после агентов с пустыми outputs.

#### Сценарий: Проверка после coder-а
- **КОГДА** coder завершается
- **ТОГДА** пайплайн ДОЛЖЕН запустить `hasGitChanges(targetDir)`
- **И** если false — остановиться с ошибкой

### Требование: by_confirm в пайплайне
Пайплайн ДОЛЖЕН поддерживать паузу после этапов с `transition: by_confirm`.

#### Сценарий: Пауза после этапа
- **КОГДА** агент с `by_confirm` завершается
- **ТОГДА** пайплайн ДОЛЖЕН показать приглашение и ждать ввод
- **И** ЕСЛИ ввод `n` — завершить пайплайн досрочно

### Требование: Loopback в пайплайне
Пайплайн ДОЛЖЕН поддерживать возврат к coder-у при REJECTED/CHANGES_REQUESTED.

#### Сценарий: Retry coder-а
- **КОГДА** reviewer вернул REJECTED
- **И** пользователь подтвердил retry
- **ТОГДА** пайплайн ДОЛЖЕН перезапустить coder с review.md как входом
- **И** reviewer ДОЛЖЕН запуститься снова после coder-а
