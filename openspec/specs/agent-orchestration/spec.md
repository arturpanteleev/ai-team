## Purpose

Спецификация определяет нормативное поведение capability `agent-orchestration`.

## Requirements
### Requirement: Порядок выполнения пайплайна
Система MUST выполнять агентов в строгом порядке, указанном в конфигурации пайплайна.

#### Scenario: Последовательное выполнение
- **КОГДА** пайплайн запускается (полный запуск)
- **ТОГДА** первый агент в конфигурации выполняется первым
- **И** каждый последующий агент выполняется только после успешного завершения предыдущего
- **И** дефолтный порядок: analyst → architect → coder → reviewer → tester → verifier → deployer

#### Scenario: Сбой пайплайна
- **КОГДА** любой агент возвращает ошибку
- **И** НЕ указан флаг `--retry-from`
- **ТОГДА** пайплайн MUST остановиться
- **И** последующие агенты MUST NOT выполняться
- **И** MUST быть сгенерирован итоговый отчёт с указанием упавшего этапа

### Requirement: Коммуникация через артефакты
Агенты MUST общаться через файлы в `.ai-team/artifacts/`.

#### Scenario: Агент читает входные данные
- **КОГДА** агент запускается
- **ТОГДА** система MUST прочитать пути его входных артефактов из `def.yaml`
- **И** проверить, что каждый входной файл/директория существует (stat)
- **И** вывести в лог: имя артефакта, полный путь, дату изменения, размер
- **И** передать содержимое файловых артефактов в runtime как `[]Artifact`
- **И** ЕСЛИ входной артефакт не найден → пайплайн MUST остановиться с ошибкой

#### Scenario: Агент записывает выходные данные
- **КОГДА** агент успешно завершается
- **ТОГДА** система MUST проверить, что его выходные артефакты существуют (stat)
- **И** вывести в лог: имя артефакта, полный путь, дату изменения, размер
- **И** добавить их к общему списку артефактов для следующего агента
- **И** ЕСЛИ выходной артефакт не создан → пайплайн MUST остановиться с ошибкой

### Requirement: Изоляция фич
Каждая фича MUST иметь своё дерево артефактов в `.ai-team/artifacts/{feature}/`.

#### Scenario: Несколько фич
- **КОГДА** пользователь запускает пайплайн для "add-auth", а позже для "add-payment"
- **ТОГДА** артефакты MUST храниться раздельно: `.ai-team/artifacts/add-auth/` и `.ai-team/artifacts/add-payment/`

### Requirement: Настройка пайплайна
Пайплайн MUST настраиваться через `.ai-team/config.yaml`.

#### Scenario: Кастомный пайплайн
- **КОГДА** пользователь указывает `pipeline: [analyst, coder, deployer]` в конфиге
- **ТОГДА** система MUST выполнить только указанных агентов в заданном порядке

### Requirement: Runtime интерфейс с передачей артефактов
Runtime MUST принимать на вход список верифицированных артефактов.

#### Scenario: Сигнатура Execute
- **КОГДА** runtime запускает агента
- **ТОГДА** Execute MUST принимать `inputs []Artifact`
- **И** Artifact содержит: Name, Path, Size, ModTime
- **И** runtime САМ решает, как передать артефакты агенту (встроить в промпт, передать флагом и т.д.)

### Requirement: Структура Artifact
`Artifact` MUST быть значением, описывающим переданный файл и его наблюдаемую метаинформацию.

#### Scenario: Поля Artifact
- `Name` — логическое имя артефакта (из `def.yaml`: "spec", "design", "review")
- `Path` — полный путь к файлу на диске
- `Size` — размер в байтах
- `ModTime` — время последнего изменения

### Requirement: Генерация HTML-отчётов в пайплайне
Пайплайн MUST генерировать HTML-отчёты после каждого этапа и итоговый сводный отчёт.

#### Scenario: Отчёт генерируется после каждого агента
- **КОГДА** агент завершает выполнение (успешно или с ошибкой)
- **ТОГДА** пайплайн MUST вызвать генерацию HTML-отчёта для этого агента
- **И** передать в отчёт: статус, время, входы, выходы, ошибку (если есть)

#### Scenario: Итоговый отчёт в конце
- **КОГДА** пайплайн завершается
- **ТОГДА** пайплайн MUST вызвать генерацию итогового сводного HTML-отчёта
- **И** отчёт MUST содержать сводку по всем этапам с навигацией

### Requirement: Уведомления в пайплайне
Пайплайн MUST вызывать Notifier после каждого завершённого этапа.

#### Scenario: Notify вызывается после каждого этапа
- **КОГДА** агент завершает выполнение
- **ТОГДА** пайплайн MUST вызвать `Notifier.Notify()` для всех зарегистрированных notifier-ов
- **И** передать StageResult: имя агента, статус, сообщение, время

### Requirement: Прогресс-бар в пайплайне
Пайплайн MUST обновлять строку состояния перед и после каждого этапа.

#### Scenario: Обновление прогресс-бара
- **КОГДА** пайплайн начинает выполнение агента
- **ТОГДА** прогресс-бар MUST отобразить имя проекта, фичи и текущего агента
- **И** после завершения агента прогресс-бар MUST обновиться

### Requirement: Ролевая конфигурация в пайплайне
Пайплайн MUST использовать индивидуальные настройки (model, effort, cli) для каждого агента.

#### Scenario: Агент запускается с индивидуальными настройками
- **КОГДА** config.yaml содержит model/cli/effort для конкретного агента
- **ТОГДА** пайплайн MUST передать эти настройки при запуске агента
- **И** MUST NOT использовать глобальные значения для этого агента

### Requirement: Retry-from в пайплайне
Пайплайн MUST поддерживать частичный запуск, начиная с указанного агента.

#### Scenario: Пропуск этапов при retry-from
- **КОГДА** пайплайн запущен с флагом `--retry-from <agent>`
- **ТОГДА** пайплайн MUST пропустить всех агентов до указанного
- **И** проверить наличие артефактов пропущенных этапов
- **И** ЕСЛИ артефакты отсутствуют — вывести ошибку и остановиться


### Requirement: Git diff guard в пайплайне
Пайплайн MUST применять mutation policy из definition, а не эвристику по имени агента или пустому outputs.

#### Scenario: Проверка после coder-а
- **КОГДА** coder завершается
- **ТОГДА** пайплайн MUST вычислить per-attempt delta относительно baseline
- **И** проверить required diff и allowed paths
- **И** если контроль не может быть выполнен — завершиться fail-closed

### Requirement: Checkpoint policy в пайплайне
Пайплайн MUST поддерживать явные `checkpoint_before` и `checkpoint_after`; legacy schema v1 `transition: by_confirm` MUST нормализоваться в interactive checkpoint.

#### Scenario: Пауза после этапа
- **КОГДА** агент с interactive или require_explicit checkpoint завершается
- **ТОГДА** пайплайн MUST показать приглашение и ждать ввод
- **И** ЕСЛИ ввод `n` — завершить пайплайн досрочно
- **И** в non-interactive режиме без явного approval MUST завершить run как stopped

### Requirement: Loopback в пайплайне
Пайплайн MUST поддерживать возврат к coder-у при REJECTED/CHANGES_REQUESTED.

#### Scenario: Retry coder-а
- **КОГДА** reviewer вернул REJECTED или CHANGES_REQUESTED
- **И** пользователь подтвердил retry
- **ТОГДА** пайплайн MUST перезапустить coder с review.md как входом
- **И** reviewer MUST запуститься снова после coder-а
- **И** попытки от coder до reviewer из прежней ветви MUST быть помечены invalidated
