# project-documentation Specification

## Purpose

Спецификация определяет нормативное поведение capability `project-documentation`:
структуру и содержание ознакомительной документации репозитория (README,
`docs/ARCHITECTURE.md`, `CONTRIBUTING.md`), чтобы она не деградировала молча в
будущих изменениях.

## Requirements
### Requirement: README как единая точка входа для onboarding
`README.md` MUST описывать назначение системы, целевую аудиторию, границы доверия, предварительные требования и полный путь поставки одной фичи без необходимости читать исходный код.

#### Scenario: Новый пользователь читает README
- **КОГДА** пользователь без предварительного контекста открывает `README.md`
- **ТОГДА** документ MUST объяснить, что такое ai-team, для какого профиля проектов он предназначен и для какого не предназначен
- **И** MUST перечислить все обязательные внешние зависимости для установки (Go toolchain, OpenCode CLI, `gh` CLI) со ссылкой на их установку

#### Scenario: Путь поставки фичи целиком
- **КОГДА** пользователь хочет провести одну фичу через весь pipeline до delivery
- **ТОГДА** README MUST содержать пошаговый walkthrough, включающий два прохода `run` (первый — до публикации canonical delivery plan, второй — с `--approve-plan <sha256>`)

### Requirement: Глоссарий обязательных терминов
`README.md` MUST включать раздел «Глоссарий», определяющий термины, использованные в Quickstart и CLI-справочнике до их первого объяснения.

#### Scenario: Термин используется до определения
- **КОГДА** Quickstart или CLI-справочник в README используют термины checkpoint, mutation scope, candidate, canonical delivery plan, verdict marker, BLOCKED или attempt/run
- **ТОГДА** README MUST содержать раздел «Глоссарий» с определением каждого из этих терминов

### Requirement: Вынесенная архитектурная документация
Глубокое описание внутреннего устройства (детерминированный контроль, evidence layout, deployer/canonical delivery plan internals, layered agent registry) MUST находиться в `docs/ARCHITECTURE.md`, а не в README.

#### Scenario: README ссылается на глубокое описание
- **КОГДА** README упоминает архитектуру системы за пределами краткого обзора конвейера
- **ТОГДА** он MUST ограничиться кратким обзором и явной ссылкой на `docs/ARCHITECTURE.md` для деталей
- **И** каждый anchor-якорь, на который README ссылается внутри `docs/ARCHITECTURE.md`, MUST соответствовать существующему заголовку в этом файле

### Requirement: Contributor-документация отдельно от agent-инструкций
Репозиторий MUST содержать `CONTRIBUTING.md`, описывающий полный OpenSpec-цикл разработки для человека-контрибьютора (explore, propose, design, specs, tasks, apply, archive) и gate rule, отдельно от `CLAUDE.md`, который адресован Claude Code как агенту.

#### Scenario: Новый контрибьютор ищет процесс разработки
- **КОГДА** контрибьютор открывает `CONTRIBUTING.md`
- **ТОГДА** документ MUST описать все стадии OpenSpec-цикла и gate rule
- **И** MUST описать make-таргеты, используемые для валидации (`make specs`, `make test`, `make verify`)

### Requirement: Автоматический regression guard для документации
Репозиторий MUST содержать автоматический тест, проверяющий, что README, `docs/ARCHITECTURE.md` и `CONTRIBUTING.md` не потеряли обязательные секции, термины глоссария и перекрёстные ссылки друг на друга.

#### Scenario: Обязательная секция удалена
- **КОГДА** из README удалена одна из обязательных секций (для кого, предварительные требования, быстрый старт, путь поставки фичи, CLI-справочник, конвейер, конфигурация, evals, глоссарий, граница безопасности, разработка)
- **ТОГДА** `go test ./docs/...` MUST завершиться с ошибкой

#### Scenario: Термин глоссария удалён
- **КОГДА** из README удалено определение одного из обязательных терминов глоссария
- **ТОГДА** `go test ./docs/...` MUST завершиться с ошибкой
